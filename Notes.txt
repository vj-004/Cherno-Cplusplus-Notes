class String{

  private:
    char* m_Buffer;
    unsigned int m_Size;
  public:
    String(const char* string){
      m_Size = strlen(string);
      m_Buffer = new char[m_Size+1]; // +1 to include the null termination character
      memcpy(m_Buffer, string,m_Size); // we can write m_Size+1 here but we are not sure if const char* string has a null termination character of not.
      m_Buffer[m_Size] = 0;
    }

    ~String(){
      delete[] m_Buffer;
    }

    char& operator[](unsigned int index){
      return m_Buffer[index];
    }

    friend std::ostream& operator<<(std::ostream& stream, const String& string);

    // lets create a copy constructor, which is called when you copy the object.
    String(const String& other) : m_Size(other.m_Size) { // By default this is provided by C++ and it just uses memcpy (shallow copy by default).
      
      // deep copy
      m_Buffer = new char[m_Size + 1]; 
      memcpy(m_Buffer, other.m_Buffer, m_Size + 1);

    }

    // String(const String& other) = delete; If you don't people to copy you objects of this class. This is how copy constructor of the unique_ptr class is implemented

}

std::ostream& operator<<(std::ostream& stream, const String& string){
  stream << string.m_Buffer;
  return stream;
}

int main(){

  String string = "Cherno";
  String second = string;

  second[2] = 'a; // Without the copy constructor, both string = "Charno" and second = "Charno". It looks like we have copied string to second but since we copied a pointer it not working like we thought

  std::cout<< string << std::endl;
  std::cout<< second << std::endl;

Without the copy constructor, when you assign second = string, it creates a copy of string which means that the char* is also copied. Both m_Buffer in string and in second point to the same memory address.
Because of this when you run the program, it will crash because when string's destructor is called it frees the m_Buffer memory and then when second's destructor is called it tries to free a memory which is already freed.

We can talk about this in terms of deep copy and shallow copy.

Always pass you objects by const reference to avoid unneseccary copying.

Dynamic Arrays in C++ (std::vector) : 

Dynamic arrays are arrays in the sense that their memory is contigous. 
When you have to resize a dynamic array. It finds a memory of a larger size and copies the original content to the new location and deletes the old memory, then keeps adding data.

Optimizing the usage of std::vector in C++:

We will basically try to optimize the part where std::vector has to resize itself, it needs to copy information from the old memory to the new memory.
This is only 1 cause of slow code but we will talk about optimizing this right now.
How can we avoid copying our objects if we are dealing with vectors (specifically object based vectors).

*Knowing your environment is the most important things with optimization.*

We need to know when copies actually happen and why they happen.

class Vertex{
  private: 
    float x,y,z;
  
  public:  
    Vertex(float x, float y, float z) : x(x), y(y), z(z) {}

    Vertex(const Vertex& vertex) : x(vertex.x),z(vertex.z),z(vertex.z) {
      std::cout<<"Copied"<<std::endl;
    }  

};


int main(){

  std::vector<Vertex> vertices;
  vertices.push_back(Vertex(1,2,3));
  vertices.push_back(Vertex(4,5,6));
  vertices.push_back(Vertex(7,8,9));

}

This outputs "Copied" 6 times.

1. When we do Vertex(1,2,3) we are creating the Vertex object in the main() functions stack frame and then we copy it into the vertices vector. 
We can avoid doing this copy by directly creating the object inside the vertices vector.

So that's 3 copies.

2. When we insert the second element, the vector needs to resize from 1 to 2, so it needs to copy the first Vertex object. simillarly when we insert the third element the vector needs to resize from 2 to 3.

Stack vs Heap memory in C++:

Stack and Heap are 2 areas which we have in the RAM of the program.
Stack is typically an area in memory which has a predefined size. Usually around 2Mb or so.
Heap also starts with a fixed value but it can grow and change as our application goes on.
The physical location of both of these memories is in the RAM.

int value = 5; // allocated an integer on the stack
int* hval = new int; // allocated on the heap. We use the new keyword to tell C++ to define it on the heap.
*hval = 5;
delete hval;

int arr[5]; // stack
int* arr = new int[5]; // heap
delete arr[];

struct Vector3{

  float x,y,z;

};

Vector3 vector; // stack
Vector3* h_vector = new Vector3();
delete h_vector;

In stack allocation we just add memory on top of each other like a stack.
It is extremely fast since it just 1 cpu cycle. We just move the stack pointer and we return the address of that stack pointer.
Once the scope of the stack allocated memory ends, the memory created in that scoped is poped.

When you allocate on the heap using the new keyword you also need delete it. Like we have discussed before if you use SMART pointers you don't need to call delete manually.
The new keyword basically calls the underlying function malloc.
When you start your application you get a certain amount of physical RAM allocated to you, and your program will maintain something called a free list. Which basically keeps track of which blocks of memory are free and where they are and all that.
And when you ask for heap memory using malloc, it will kind of be able to go through the free list and find a memory that is at least as big is the size of the memory you have requested.
If you ask for more memory that is actually in that free list then the program will ask the OS for more memory which is very expensive.
With the heap there is whole lot of book keeping which is also done. You also need to delete the memory.

Essentially the point is that, stack is super fast and heap is slow.
Always allocated on the stack if possible.
Unless you want to allocated a large amount of memory (which can't be done on the stack), use the heap.
The performance difference come strictly from the allocation. So if you were to theoretically pre-allocated 4GB of memory before you ran you program on the heap. And then you were to "heap allocated" from that pre-allocated block of memory.
It would basically be the same as stack, you would get a few CPU cache misses but there are not enough of them to actually matter.

Casting in C++:

Type casting in any kind of conversion we want to do within the type system provided to us in C++.
You need to specifically do type casting until and unless it is a implicit type cast (implicit conversion).

There is C styled cast and there is C++ styled cast.

int a = 5;
double b = a;

This is implicit conversion

double b = 5.25;
int a = value; 

This is also an implicit conversion
If we want to do this explicitly, we can do,

double c = (int)b + 5.3; // This would result in 10.3
This is a C styled cast.

double s = static_cast<int>(value) + 5.3;
This is a C++ styled cast. It is static_cast<the value which we want to cast it into> (variable/value).

There are many casts in C++,
1. static cast
2. reinterpret cast
3. dynamic cast
4. const cast

They do not do anything that C styled cast cannot do, ofcourse they do some additional things but the result of what you get is not different from what you would get in C styled cast.
These cast are not adding any new functionality, there are only some adding some syntax sugar.

For example dynamic cast, will actually perform a check and it may return null if the type conversion wasn't successfull. It does run time checking.
It does some extra things which also means it also kinda slows you down.

static_cast also have some checks of its own, like if the type conversion can never take place, it will give you an error when you are writing code.
reinterpret_cast is like type punning.

With this it adds compile time checking.

Dynamic cast is tied very closely to Run Time Type Information (RTTI).


Precomiled Headers in C++:

They give you an opportunity to grab a bunch of header files and covert them into a compiled format which the compiler can then use instead of compiling the header files again and again.
With out precompiled headers everytime you make the smallest change to your C++ program all the header files need to compiled again and this wastes a lot of time in compiling.
If you have mulitiple C++ file in your project and they all include vector then you will have to compile the vecter.h header file those many number of times. This is wasted time during compiling.

Why to not use precompiled headers? 
The precompiled headers is essentially a header file which contains a bunch of header files. And because of that a lot of people might be like I can put every thing in my project into my PCH(pre compiled header).
    If I do that I will get really fast build times right?
      Yes but, if you put stuff in your PCH which actually changes, then obviously your precompiled header has to be rebuilt and that takes time and it could potentially increase your compilation time from before.
      So don't put frequently changing files in the PCH.

    It is mostly used for code/libraries that isn't yours. Because you won't be modifing these libraries. Put all of this in your precompiled header.
    
    There is one more thing PCH can do, because you shove everything in your PCH, it can potentially hide what is actually being used.
    For example you have used PCH and you are writing a CPP file. Now you/other wants to take this file and use it somewhere else but now you don't know which all header files this CPP file actually uses.

Different IDEs' and different compilation methods have differnt ways on implementing precompiled headers.

Dynamic Casting in C++:

Dynamic casting gives us kind of a safety net when we do a particular type of casting.
It is a C++ styled cast which only works in C++, and it does some extra work to make sure that the way that we actually cast is actually a valid cast.

Dynamic Cast only works with polymorphic class types, that means that we need to have virtual function table in the Entity class.

class Entity{
  virtual void Anything() {}
};

class Player : public Entity{

};

class Enemy : public Entity{

};

int main(){

  Player* player = new Player(); // player is of type Player, Entity and it also of a type "typeless". 
  Entity* actuallyPlayer = player;

  // Player* p = actuallyPlayer;  This will give an error because e could have very easily also been a Enemy. So we need to type case.
  Player* p = (Player*)actuallyPlayer;

  Entity* actuallyEnemy = new Enemy();
  Player* p1 = (*actuallyEnemy)e1; // This would work but this is potentially dangerous because if we try to access something which is only defined for the Player class, using *p1 would crash our program or make it do some really weired things which we don't wanna do.

  Player* p2 = dynamic_cast<Player*>(actuallyEnemy); // This should fail. p3 will be null.
  Player* p3 = dynamic_cast<Player*>(actuallyPlayer);

}

How does it know that the entity which I have is actually a player and not an enemy? 
The way that it does that it is it acutally stores the run time type information (RTTI) about all of our types, 
This adds overhead but it lets you do things like dynamic casting.
The dynamic cast itself also takes extra overhead because it needs to run some checks.
Dynamic casts require RTTI to be ON whenever they are to be used.

Benchmarking in C++:

class Timer{

public:
  Timer(){

    m_StartTimePoint = std::chrono::high_resolution_clock::now();

  }

  ~Timer(){
    Stop();
  }


  void Stop(){
    auto endTimePoint = std::chrono::high_resolution_clock::now();
    auto start = std::chrono::time_point_cast<std::chrono::microseconds>(m_StartTimePoint).time_since_epoch().count();
    auto end = std::chrono::time_point_cast<std::chrono::microseconds>(endTimePoint).time_since_epoch().count();
    
    auto duration = end - start;
    double ms = duration * 0.001;

    std::cout << duration << "us (" << ms << "ms)\n";
  }

private:
  std::chrono::time_point<std::chrono::high_resolution_clock> m_StartTime;
};

How to make C++ run FASTER (with std::async):

You should remember you can't mindlessly break your program into multiple thread. You need to check which part of the program can actually be processed separately and break the program accordingly.

#include <future>

void LongTask(){
  //code
}

void DoTask(std::vector<std::future<void>>& m_Futures){
  // lets say I want to run the LongTask function say 20 times. All the function calls are independent of each other so I can call them in separate threads.
  // so to do that,

  for(int i=0;i<20;i++){
    /* 
      we need to set what kind of job this will be, it is important to set the launch type to be async. 
      if instead we set it to defered or defered+async then it may not be actually done on a separate thread. 
      C++ will choose that on the current work load.

      std::async(std::launch::async, LongTask); 
      
      std::async actually returns an std::future and it is really important that we hang on to that std::future.
      Because if we don't hang on to that std::futures, it will be destroyed and upon destruction it needs to ensure that the async function needs
      to be finished that would basically mean that it won't be parallel at all.

      Upon completing one iteration of this for loop, 
        the LongTask function will actually in the destructor of std::future wait for the LongTask method to be finished.
        So basically we need to store this as a member varaible. 
    */

    m_Futures.push_back(std::async(std::launch::async, LongTask));
    
  }
}

int main(){

  
  std::vector<std::future<void>> m_Futures; // here void is basically the return type of the function which will be "stored" in std::future.
  DoTask(m_Futures);

}
