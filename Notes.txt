Inheritance: If a class B inherits from another class A, then class B also "is of type A". This means that class B can also used as class A while passing in functions because class B is garenteed to have all the (public) functionality that A has.

A function is made virtual in a class if a derived class may override it.
Virtual Functions: they introduce something called dynamic dispatch which compilers implement using something called a V-table. It is a table which maps all the virtual function inside our base class so that we can map them to the correct overwritten function in run time.
Virtual Functions aren't free, there are 2 run time costs associated with it:
1) Initial memory to store the v-table
2) Every time we call a virtual function, we need to look through the table to check to which function we acutally map
(cost is negligible until its not)
When we declare a PURE VIRTUAL function, we are forcing the derived class to implement it.

Member Initializer Lists in C++ : 

class Entity{

  private: 
    string m_Name;
    int m_Score;

  public:
    Entity()
      : m_Name("Vayun"), m_Score(0)
    {

    }

    Entity(const string& name) : m_Name(name)
    {

    }
}

Instead of initializing them inside the constructor function, you can put a " : " and then Initialize the member variables before starting the function definaiton, this is called Member Initializer List (MIL)

The initialization should be in the order in which they are declared. (The member variables are initialized in the same order they are declared if you don't follow the order while initializing them as a MIL).

What difference does it make?
1) One of the difference is that it helps keeps your code clean, you can write all you initializations outside the function in a line, this way you can clearly see what the function is actually doing.
2) There is also a difference functionally and this difference applies to classes specifically,  

class Example{
  public:
    Example(){
      cout<<"created an example\n";
    }
    Example(int x){
      cout<<"created an example with "<<x<<"!"<<'\n';
    }
}

class Entity{
  Example m_Example;
  Entity(){  
    m_Example = Example(8); 
  }
}

Here you will be calling the Example class twice, the output will be:
created an example
created an example with 8!

When you are declaring the example class, the default constructor is called there. Thus we have created 2 objects(of Example) instead of 1, so there is a performance issue

while if we did,


class Entity{
  Example m_Example;
  Entity() : m_Example(Example(8))
  {  
  }
}

or 

class Entity{
  Example m_Example;
  Entity() : m_Example(8)
  {  
  }
}

In both these cases you will crete only one instance of the example class

* You should be using member initializer lists everywhere


How to CREATE/INSTANTIATE OBJECTS in C++:

We have 2 choices to do this and the difference between the 2 choices is where the memory comes from.
When ever we create an object in C++, it needs to occupy some memory. Even an object with 0 member variables and 0 member functions still occupies at least 1 byte of space.

We as a programmer have the ability to create objects either on the stack or on the heap, both have their own functional differences.
For example stack objects have an automatic lifespan, thier life time is controlled by the scope they are declared in.

Stack:

class Entity{
  string m_Name;
  Entity(): m_Name("Unknown"){  
  }
  Entity(const string& name): m_Name(name){  
  }
}

Entity entity_a; // (In languages such as Java you would get a NULL pointer exception, but in C++ this is totally valid)
Entity entity_b("Vayun");

Entity entity_c = Entity("Vayun");

entity_a, entity_b, entity_c are valid objects of the class Entity created in the stack memory. 

*Whenever possible always use the stack space. ( This is the fastest / "most managed" way in C++ to instantiate objects) 

Heap:

Entity* entity = new Entity("Vayun")

The main difference here is not the pointer which a lot of a people mistake, its actually the "new" keyword.
Allocating on the Heap is bad for performance.
When you allocate memory on the heap, you need to manually free the memory you have allocated. (it is also automatically deleted when the program ends)

delete entity;

If you the "new" keyword to create something, it is allocated on the heap and you need to use the "delete" keyword.

*Create objects/anything on the stack unless you absolutely can't.

Implicit Conversion and the Explicit Keyword in C++: 

class Entity{
  private:
    std::string m_Name;
    int m_Age;
  public:
    Entity(const std::string& name) m_Name(name) m_Age(-1): {} 
    Entity(int age) m_Name("Unknown") m_Age(age): {} 
}

void Print(Entity& entity){
  // Printing something
}

int main(){

  Entity a("Vayun");
  Entity b(22);

  // This works, but what you can also do is

  Entity a = "Vayun"; // C++ Implicitly converts this too match the type in Entity Constructor if possible.
  Entity b = 22; // C+ + Implicitly converts the integer and the string to an Entity type based on the constructor.

  Print(22); // This also works because C++ can find the constructor for Entity class which takes an integer so it Implicitly converts it to an Entity type
  // *Print("Vayun");* You would expect this also to work but this gives an error, becasue the Entity constructor expects a const std::string but this is a const char[6]
  // For this to work C++ would have to do 2 implicit conversions, one from const char[] into a string and then from string to entity, but C++ is allowed to do only 1 implicit conversion

  Print(std::string("Vayun")); // This would work becasue C++ needs to do only 1 implicit conversion
  Print(Entity("Vayun")); // This also works.
}

If we define print as

void Print(const Entity& entity){
  //Printing
}

Then C++ can do multiple implicit conversions, so 
Print("Vayun") would work in this case;

C++ does NOT allow multiple implicit conversions for function arguments when binding to a non-const reference.

Entity a = "Vayun";
This involes a copy constructor, so even though "Vayun" is a const char[], it still converts it into a const std::string

C++ rule:
In copy initialization, the compiler is allowed to:
  Use a constructor
    And perform conversions to match that constructor’s parameter

There is something called the explicit keyword which prevents implicit conversion, you put explicit keyword before a constructor

class Entity{
  private:
    std::string m_Name;
    int m_Age;
  public:
    explicit Entity(const std::string& name) m_Name(name) m_Age(-1): {} 
    Entity(int age) m_Name("Unknown") m_Age(age): {} 
}

Entity a = "Vayun"; // This would fail
Entity b = 22; // This would not fail becasue the constructor for this is not explicit.
