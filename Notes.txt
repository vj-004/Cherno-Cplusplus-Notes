Class: Syntaxical sugar, used to make the life of programmers easy (helps us write clean code)

Methods in a class/struct: non-static methods in a class get a instance of the class as parameter by default, but static methods don't get this parameter, so they cannot access non-static members of the class.

Constructors: If you don't want anybody to be able to instanciate a class, you can create an empty constructor as a private method or you can write "class_name() = delete;" under public and you can't instanciate your class anymore.

Inheritance: If a class B inherits from another class A, then class B also "is of type A". This means that class B can also used as class A while passing in functions because class B is garenteed to have all the (public) functionality that A has.

Virtual Functions: they introduce something called dynamic dispatch which compilers implement using something called a V-table. It is a table which maps all the virtual function inside our base class so that we can map them to the correct overwritten function in run time.

Virtual Functions aren't free, there are 2 run time costs associated with it:
1) Initial memory to store the v-table
2) Every time we call a virtual function, we need to look through the table to check to which function we acutally map
(cost is negligible until its not)

Arrays:
2 ways to define it:
    1) Stack: int a[5]
    2) Heap: int* a = new int[5]    

The array declared in stack will be deleted when the function ends
but the array declared in the heap will be deleted when the program terminates unless you delete it before that

You can delete the array in heap by doing :  delete[] a

Creating an array on the heap causes a memory indirection which means that the variable points to another pointer which points to the data which is a performance hit

If you create an array in the heap, then you can "never" retreive the size of the array. Although when you delete this array the compiler needs to know the size of the array so there should be some way to get it right, the things is that it is compiler dependent for example some compilers may store it at a "-1" index of the array, its just not reliable enough to use it.

So you can "never" retreive the size of the array allocated in heap

When you allocate an array on the stack its size must be a compiled time known constant otherwise you will get an error* Cherno: "There are ways around this cuz its C++ but yeah" 

In C++ 11 there is a standard array which is allocated in heap and you can do variable_name.size() to get the size of the array.
eg: std::array<int, 5> arr;
int size_arr = arr.size();
(It does have its own overhead though)

CONST in C++:

It is a like a promise, it is a promise that something will be constant. But like in real life, there are ways to break this promise
It also help us enforce certain rules of the developer who will working with the code

int* a = new int;
Here you can change the contents of a and also you can change the address, the variable a is pointing to

const int* a = new int;
Here you cannot change the contents of a but you can change the address, the variable a is pointing to

int* const a = new int;
Here you can change the contents of a but you cannot change the address. the variable a is pointing to

int const* a = new int;

is the same as const int* a = new int;

const int* const a = new int;
Here you cannot change the contents of a and you also cannot change the address, the variable a is pointing to

Mutable:

When you are dealing with a lambda function, sometimes you pass a variable by value, if you do this you cannot modify it, but what if you do want to modify a variable, in that case you make the lambda function to be mutable.

*We will learn lambda functions later
* This is a rare use case of the mutable keyword ( not usually used in code)
int x = 8;
auto f = [=](){
  x++;  // this will give an error
  cout << x << endl;
}


int x = 8;
auto f = [=](){ mutable
  x++;  // this will not give an error
  cout << x << endl;
}


Member Initializer Lists in C++ : 

class Entity{

  private: 
    string m_Name;
    int m_Score;

  public:
    Entity()
      : m_Name("Vayun"), m_Score(0)
    {

    }

    Entity(const string& name) : m_Name(name)
    {

    }
}

Instead of initializing them inside the constructor function, you can put a " : " and then Initialize the member variables before starting the function definaiton, this is called Member Initializer List (MIL)

The initialization should be in the order in which they are declared. (The member variables are initialized in the same order they are declared if you don't follow the order while initializing them as a MIL).

What difference does it make?
1) One of the difference is that it helps keeps your code clean, you can write all you initializations outside the function in a line, this way you can clearly see what the function is actually doing.
2) There is also a difference functionally and this difference applies to classes specifically,  

class Example{
  public:
    Example(){
      cout<<"created an example\n";
    }
    Example(int x){
      cout<<"created an example with "<<x<<"!"<<'\n';
    }
}

class Entity{
  Example m_Example;
  Entity(){  
    m_Example = Example(8); 
  }
}

Here you will be calling the Example class twice, the output will be:
created an example
created an example with 8!

When you are declaring the example class, the default constructor is called there. Thus we have created 2 objects(of Example) instead of 1, so there is a performance issue

while if we did,


class Entity{
  Example m_Example;
  Entity() : m_Example(Example(8))
  {  
  }
}

or 

class Entity{
  Example m_Example;
  Entity() : m_Example(8)
  {  
  }
}

In both these cases you will crete only one instance of the example class

* You should be using member initializer lists everywhere
